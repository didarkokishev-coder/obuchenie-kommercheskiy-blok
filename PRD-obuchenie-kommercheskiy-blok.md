# PRD: Обучение и оценка менеджеров по продажам — «Стандарт сервиса и техника продаж»

**Компания:** АО «Казахтелеком»
**Блок:** Коммерческий (менеджеры по продажам)
**Тип документа:** Product Requirements Document
**Статус:** Draft / MVP реализован

---

## 1. Контекст и проблема

Менеджеры по продажам коммерческого блока нерегулярно и неравномерно владеют стандартом сервиса и техникой продаж, что приводит к:
- разному качеству обслуживания клиентов в зависимости от менеджера;
- необходимости готовиться к ежемесячному тестированию сотрудников без единого источника материала;
- отсутствию у ментора/администратора удобного способа увидеть, кто из сотрудников прошёл самоподготовку и с каким результатом.

Требуется короткий (≈20 минут) самостоятельный обучающий модуль, который одновременно служит подготовкой к ежемесячному тестированию и даёт администратору видимость результатов по каждому сотруднику.

---

## 2. Цель продукта

Дать менеджерам по продажам самостоятельный, короткий и практичный курс по стандарту сервиса и технике продаж, и дать администратору (ментору) возможность видеть, кто прошёл обучение и с каким результатом — без развёртывания отдельной LMS или бэкенда.

### Критерии успеха (Success Metrics)
- Прохождение курса занимает ≤ 20 минут.
- ≥ 80% менеджеров проходят чек-лист полностью.
- Средний балл финального теста ≥ 75%.
- Администратор может получить сводку по всем сотрудникам без ручного сведения таблиц (кроме первичной настройки источника данных).

---

## 3. Пользователи и роли

| Роль | Кто это | Что делает в продукте |
|---|---|---|
| **Менеджер по продажам** | Новый или действующий сотрудник коммерческого блока | Проходит обучение, отмечает чек-лист, проходит тест, отправляет результат |
| **Администратор / ментор** | Руководитель группы, наставник | Заходит в закрытую панель, видит сводную статистику и таблицу результатов по сотрудникам |

---

## 4. Состав продукта

Продукт — **один HTML-файл**, содержащий два режима (view), переключаемых на клиенте:

1. **Обучающая часть** (публичная, открывается по умолчанию)
2. **Админ-панель** (защищена кодом доступа, открывается через плавающую кнопку)

Хостинг — статичная страница на внутреннем портале/интранете компании. Отдельного сервера/бэкенда нет.

---

## 5. Функциональные требования

### 5.1 Обучающая часть

**Прогресс**
- Закреплённая сверху шкала прогресса из 4 сегментов, заполняется по мере прокрутки.
- Индикатор оставшегося времени («~15 мин осталось» и т.д.), пересчитывается по scroll-позиции.

**Блок 0 — Приветствие**
- Заголовок, описание цели курса, ориентировочное время (4 блока × ~20 минут).
- Поле **ФИО** (обязательное для последующей отправки результата).
- Кнопка «Начать обучение» — скролл к первому блоку.

**Блок 1 — Чек-лист обслуживания**
- 7 пунктов стандарта сервиса, каждый кликабелен (toggle done/undone).
- Счётчик «Отмечено X из 7», визуальная индикация завершения.

**Блок 2 — Техника продаж**
- 5 шагов техники продаж (контакт → потребность → презентация → возражения → закрытие) в виде аккордеона.
- У каждого шага: краткое описание + пример фразы.

**Блок 3 — Удержание клиентов**
- 4 карточки: проактивная забота, персональные предложения, работа с недовольством, программа лояльности.

**Блок 4 — Финальный тест**
- 4 вопроса с выбором одного варианта из четырёх.
- После выбора — немедленная проверка (правильный вариант подсвечивается зелёным, неверный выбор пользователя — красным), варианты блокируются.
- Подсчёт правильных ответов в реальном времени.

**Результаты и отправка**
- Карточка с итогом: чек-лист (X из 7), тест (X из 4), время на странице (в минутах, считается от загрузки страницы).
- Кнопка «Отправить результат» — передаёт данные администратору (см. раздел 6, механизм передачи данных).
- Валидация: без заполненного ФИО отправка блокируется, пользователя возвращает к полю ФИО.

### 5.2 Админ-панель

**Вход**
- Плавающая кнопка (иконка замка) в правом нижнем углу экрана, всегда видна в обучающей части.
- Дублирующая текстовая ссылка в футере страницы.
- При клике — системное окно браузера (`window.prompt`) с запросом кода доступа.
- Неверный код — сообщение об ошибке (`alert`), доступ не открывается.
- Код доступа хранится в теле страницы в виде константы (не является криптографической защитой — служит фильтром от случайных заходов).

**Содержимое панели**
- Сводная статистика (4 карточки): количество прошедших обучение, средний балл теста (%), доля сотрудников с полностью пройденным чек-листом (%), среднее время прохождения (мин).
- Таблица результатов по каждому сотруднику: дата, ФИО, чек-лист (X из 7 + цветовой индикатор), тест (X из 4 + цветовой индикатор), время.
- Поиск по ФИО (живой фильтр).
- Сортировка по каждому столбцу таблицы (клик по заголовку).
- Кнопка «Обновить» — повторный запрос данных из источника.
- Ссылка «К обучению» — возврат в обучающую часть.

**Цветовая индикация результата (pill)**
- ≥ 80% — зелёный («good»)
- 50–79% — жёлтый («mid»)
- < 50% — красный («bad»)

## 6. Передача и хранение данных

Поскольку у продукта нет собственного бэкенда, данные передаются и хранятся через внешние сервисы Google (бесплатно, без разработки сервера):

1. **Запись результата** — при отправке со страницы обучения данные (ФИО, чек-лист, тест, время, дата) отправляются POST-запросом на веб-приложение **Google Apps Script**, которое дописывает строку в **Google Таблицу**.
2. **Чтение результатов** — админ-панель запрашивает ту же таблицу, опубликованную в формате **CSV** (Файл → Опубликовать в интернете), парсит CSV на клиенте и строит таблицу/статистику.

**Требуемая настройка перед запуском (разово, вне кода продукта):**
- Создать Google Таблицу с заголовками: `Дата | Имя | Чек-лист | Квиз | Время`.
- Подключить Apps Script (`doPost`) и опубликовать как веб-приложение (Execute as: Me, Access: Anyone).
- Опубликовать таблицу в формате CSV.
- Вставить обе ссылки и код доступа в константы `ENDPOINT_URL`, `SHEET_CSV_URL`, `ADMIN_PASSCODE` в коде страницы.

**Альтернатива (рассмотрена, не выбрана как основной вариант):** отправка результата по email (`mailto:`) с ручной вставкой текста писем в админ-панель для парсинга. Требует меньше настройки, но не даёт автоматической сводки без участия администратора и не хранит историю между сессиями браузера.

## 7. Нефункциональные требования

- **Адаптивность:** корректное отображение на мобильных экранах (проверено вплоть до ~360px ширины).
- **Автономность:** один HTML-файл, без сборки, без npm/бандлера; шрифты подключаются через Google Fonts CDN.
- **Производительность:** без тяжёлых зависимостей, весь JS — vanilla, без фреймворков.
- **Стиль:** корпоративный, единая система токенов (цвета, типографика Golos Text + Inter), последовательная визуальная айдентика между обучающей частью и админ-панелью.
- **Доступность:** поддержка `prefers-reduced-motion`, видимый фокус на интерактивных элементах.
- **Устойчивость к ошибкам ввода:** отправка результата и вход в админ-панель не должны молча падать — пользователь всегда видит статус (успех/ошибка).

---

## 8. Ограничения и вне рамок (Out of scope)

- Нет полноценной аутентификации/ролевой модели — доступ в админку основан на общем коде доступа, а не на персональных учётных записях.
- Нет постоянного хранилища на стороне клиента (localStorage/sessionStorage не используются) — данные админ-панели актуальны только на момент запроса к источнику.
- Нет автоматических email/push-уведомлений администратору о новых результатах (кроме варианта с `mailto`, который не является основным).
- Нет управления содержимым курса через UI — правки контента вносятся напрямую в HTML.
- Нет мультиязычности — контент только на русском.
- Нет интеграции с существующими HR/LMS-системами компании (при наличии таковых — требует отдельного обсуждения).

## 9. Риски и открытые вопросы

| Риск / вопрос | Комментарий |
|---|---|
| Код доступа виден в исходном коде страницы | Приемлемо как «мягкий» барьер; для строгой защиты нужна авторизация через корпоративный портал |
| Google Apps Script / Google Таблицы — использование внешнего облачного сервиса для внутренних данных компании | Нужно согласовать с ИБ/комплаенс, если результаты обучения считаются чувствительными данными |
| Отсутствие персистентности данных в самой админ-панели (только источник — Google Таблица) | При недоступности интернета на момент открытия панели данные не подгрузятся |
| Нет объединения с существующей системой ежемесячного тестирования | Курс задуман как подготовка к тестированию, но не заменяет и не интегрирован с самим тестированием |

## 10. История итераций (Changelog)

1. **v0.1** — Статичная одностраничная лендинговая версия обучения без сбора результатов (4 блока, прогресс-бар, интерактивный чек-лист и квиз в формате «показать ответ»).
2. **v0.2** — Добавлено поле ФИО, финальный тест переработан в проверяемый (правильно/неправильно), добавлен подсчёт времени и чек-листа, добавлена отправка результата через Google Apps Script + Google Таблицу.
3. **v0.3** — Обучение и админ-панель объединены в один файл с переключением по URL-параметру (`?admin=1`) и кодом доступа.
4. **v0.4 (рассмотрено, откачено)** — Временный переход на сбор результатов через email (`mailto`) и ручную вставку писем в админку — отклонён администратором, возвращена версия с Google Таблицей.
5. **v0.5 (текущая)** — Улучшена находимость входа в админ-панель (плавающая кнопка), вход в панель переведён на нативное окно браузера (`window.prompt`) вместо кастомного поля ввода — для устранения проблем с фокусом/автозаполнением в разных браузерах.
